复习一下，事务的ACID特性：

  • Atomicity（原子性）：事务是不可分割的单元，对于一个事务，所有的操作要么全部成功，要么全部失败。

  • Consistency（一致性）：事务前后，数据必须保持一致，如A向B转账，A和B的总金额就是个一致性状态。

  • Isolation（隔离性）：多个并发事务之间相互隔离，操作不能互相干扰。常用的方式是加锁。

  • Durablity（持久性）：事务提交后，更改是永久的。

事务并发的脏读、幻读问题、事务隔离级别：待复习。

两阶段提交协议（Two Phase Commit，2PC）：

  • Client向Coordinator（中心协调节点）发起一个事务请求；

  • Coordinator向N个Partcipant（参与者节点）发送PrepareRequest（事务预处理请求）；

  • Partcipant收到PrepareRequest后，如果可以处理，则执行本地事务，但不提交，而给Coordinator回复YES，如果不能处理，回复NO；

  • Coordinator收到所有的Partcipant的YES后，再向Partcipant发送提交请求，Partcipant再提交本地事务；

  • 否则，Coordinator向所有的Partcipant发起回滚，Partcipant执行回滚后返回确认。

问题：

  • Coordinator发送PrepareRequest后，如果宕机，Partcipant事务会一直挂起直到超时；

  • Partcipant直到收到提交请求后，才会释放，这个过程是阻塞的；

  • 如果因为网络原因，只有部分Partcipant收到了提交请求，就会导致不一致的问题。

三阶段提交协议（Three Phase Commit，3PC）：

  • CanCommit：Client向Coordinator发起一个事务请求，Coordinator向N个Partcipant发送CanCommit询问，是否可以完成事务，比如尝试获取数据库锁，可以则回复YES，否则NO；

  • PreCommit：

    • 如果所有的Partcipant都返回YES，Coordinator则向所有Partcipant发送PreCommit请求，Partcipant收到后执行事务，但不提交，并记录Undo和Redo信息到日志，向Coordinator回复YES，同时等待指令；

    • 如果有Partcipant回复NO，或Coordinator等待Partcipant回复超时，则向所有Partcipant发送Abort请求，中断事务；

  • DoCommit：

    • 如果所有的Partcipant都返回YES，Coordinator则向所有Partcipant发送DoCommit请求，Partcipant收到后提交事务，向Coordinator回复YES，Coordinator收到所有的Partcipant的YES后完成事务；

    • 如果有Partcipant回复NO，或Coordinator等待Partcipant回复超时，则向所有Partcipant发送Abort请求，中断事务，Partcipant收到后根据Undo信息执行回滚，向Coordinator回复YES。

在DoCommit阶段，假如因为网络原因或者Coordinator宕机，Partcipant没有收到DoCommit或Abort请求，都会在等待超时之后，继续进行事务提交，所以可能会出现不一致的情况。

补偿事务（Try Confirm Cancel，TCC）：

对每个事务分支操作都要注册一个对应的确认和补偿（撤销）操作（Try Confirm Cancel）。

  • Try：主要是对业务系统做检测及资源预留；

  • Confirm：确认执行业务操作，要求Try成功Confirm一定要能成功；

  • Cancel：取消执行业务操作。

以扣钱为例，在接入TCC前，对A账户的扣钱，只需一条更新账户余额的SQL。

在接入TCC之后，需要将原来一步完成的扣钱操作，拆成两阶段，实现成三个方法，并且保证一阶段Try成功的话二阶段Confirm一定能成功。

Try就是检查账户余额是否充足，锁定转账资金（加个锁或打个标记），此时账号A余额虽然还是100，但是其中30元已经被冻结了，不能被其他事务使用。

Confirm执行真正的扣钱操作，使用Try阶段锁定的资金，执行扣款，则账号A余额变成70元。

如果二阶段是回滚的话，需要在Cancel方法内释放一阶段Try冻结的30元，使账号A回到初始状态，100元全部可用。

在分布式环境下，就是Coordinator依次调用所有Partcipant的Try方法，所有都正常的话，再依次调用Partcipant的Confirm方法，否则调用Cancel方法。

假设某个Partcipant意外宕机然后重启，可以从持久化日志中恢复运行的阶段和状态。

假设某个Partcipant的Cancel或Confirm方法调用失败，则Coordinator不断重试，直到成功。

RocketMQ的事务消息

  • Producer执行本地事务之前，向Broker发送PREPARED消息，并且消息发送成功（确认），声明事务开始，但此消息不会给Consumer消费；

  • Producer执行本地事务，根据事务结果Commit或Rollback本地事务；

  • Producer根据事务结果向Broker发送COMMIT或ROLLBACK消息，更新消息状态，并结束事务，发送是单向的，不保证成功；

  • Broker收到消息后，如果是COMMIT消息，则可以投递给Consumer消费，如果是ROLLBACK消息，则废弃消息；

  • 对于Producer执行事务超时，或者发送事务状态消息失败或丢失等情况，Broker定时向Producer回查事务状态，并更新消息状态。

因为MQ是异步的，所以这里是最终一致，而不是强一致。

Put simply，事务消息的用处就是Producer执行本地事务成功提交后，Consumer再从Broker拉取消息，然后继续执行本地事务，回复消费确认。

那如果Consumer执行本地事务失败后，如何回滚Producer的事务呢？官方的意见是，自己手动解决，框架不提供此功能。

本地事务状态

  • COMMIT：成功

  • ROLLBACK：失败

  • UNKNOWN：未知，可能是事务正在执行，也可能是成功或失败

    public TransactionSendResult sendMessageInTransaction(final Message msg, final LocalTransactionExecuter localTransactionExecuter, final Object arg) throws MQClientException {
        // 检查TransactionListener
        TransactionListener transactionListener = getCheckListener();
        if (null == localTransactionExecuter && null == transactionListener) {
            // ...
        }
        // ...
        SendResult sendResult = null;
        // 设置为PREPARED消息
        MessageAccessor.putProperty(msg, MessageConst.PROPERTY_TRANSACTION_PREPARED, "true");
        // 设置ProducerGroup，用于事务回查
        MessageAccessor.putProperty(msg, MessageConst.PROPERTY_PRODUCER_GROUP, this.defaultMQProducer.getProducerGroup());
        try {
            // 同步发送消息
            sendResult = this.send(msg);
        } catch (Exception e) {
            // ...
        }
        // 本地事务初始状态，UNKNOW
        LocalTransactionState localTransactionState = LocalTransactionState.UNKNOW;
        Throwable localException = null;
        // 发送结果
        switch (sendResult.getSendStatus()) {
            // 发送成功，则同步Master已刷盘，异步Master可能未刷盘
            case SEND_OK: {
                try {
                    // ...
                    String transactionId = msg.getProperty(MessageConst.PROPERTY_UNIQ_CLIENT_MESSAGE_ID_KEYIDX);
                    if (null != transactionId && !"".equals(transactionId)) {
                        msg.setTransactionId(transactionId);
                    }
                    if (null != localTransactionExecuter) {
                        localTransactionState = localTransactionExecuter.executeLocalTransactionBranch(msg, arg);
                    } else if (transactionListener != null) {
                        // 通过TransactionListener执行本地事务，返回本地事务状态
                        localTransactionState = transactionListener.executeLocalTransaction(msg, arg);
                    }
                    if (null == localTransactionState) {
                        localTransactionState = LocalTransactionState.UNKNOW;
                    }
                    // ...
                } catch (Throwable e) {
                    // ...
                }
            }
            break;
            // 发送成功，但是Broker持久化消息超时
            case FLUSH_DISK_TIMEOUT:
                // 发送成功，但是同步Master将消息同步到Slave超时
            case FLUSH_SLAVE_TIMEOUT:
                // 发送成功，但是同步Master未找到可用Slave
            case SLAVE_NOT_AVAILABLE:
                // 都要回滚
                localTransactionState = LocalTransactionState.ROLLBACK_MESSAGE;
                break;
            default:
                break;
        }
        try {
            // 结束事务，回滚或提交，或未知
            this.endTransaction(sendResult, localTransactionState, localException);
        } catch (Exception e) {
            // ...
        }
        TransactionSendResult transactionSendResult = new TransactionSendResult();
        // ...
        return transactionSendResult;
    }

DefaultMQProducerImpl#endTransaction

    public void endTransaction(final SendResult sendResult, final LocalTransactionState localTransactionState, final Throwable localException) throws RemotingException, MQBrokerException, InterruptedException, UnknownHostException {
        final MessageId id;
        if (sendResult.getOffsetMsgId() != null) {
            id = MessageDecoder.decodeMessageId(sendResult.getOffsetMsgId());
        } else {
            id = MessageDecoder.decodeMessageId(sendResult.getMsgId());
        }
        // Broker返回的事务Id
        String transactionId = sendResult.getTransactionId();
        // 接收PREPARED消息的Broker地址
        final String brokerAddr = this.mQClientFactory.findBrokerAddressInPublish(sendResult.getMessageQueue().getBrokerName());
        EndTransactionRequestHeader requestHeader = new EndTransactionRequestHeader();
        // 事务Id
        requestHeader.setTransactionId(transactionId);
        // 消息在CommitLog的偏移
        requestHeader.setCommitLogOffset(id.getOffset());
        // 本地事务状态
        switch (localTransactionState) {
            // 提交
            case COMMIT_MESSAGE:
                requestHeader.setCommitOrRollback(MessageSysFlag.TRANSACTION_COMMIT_TYPE);
                break;
            // 回滚
            case ROLLBACK_MESSAGE:
                requestHeader.setCommitOrRollback(MessageSysFlag.TRANSACTION_ROLLBACK_TYPE);
                break;
            // 事务状态不明
            case UNKNOW:
                requestHeader.setCommitOrRollback(MessageSysFlag.TRANSACTION_NOT_TYPE);
                break;
            default:
                break;
        }
        requestHeader.setProducerGroup(this.defaultMQProducer.getProducerGroup());
        // Queue的偏移
        requestHeader.setTranStateTableOffset(sendResult.getQueueOffset());
        requestHeader.setMsgId(sendResult.getMsgId());
        // ...
        // 单向发送结束事务消息，END_TRANSACTION请求，后续由Broker负责回查
        this.mQClientFactory.getMQClientAPIImpl().endTransactionOneway(brokerAddr, requestHeader, remark, this.defaultMQProducer.getSendMsgTimeout());
    }

TransactionalMessageBridge#parseHalfMessageInner

    private MessageExtBrokerInner parseHalfMessageInner(MessageExtBrokerInner msgInner) {
        // 备份原Topic
        MessageAccessor.putProperty(msgInner, MessageConst.PROPERTY_REAL_TOPIC, msgInner.getTopic());
        // 备份原QueueId
        MessageAccessor.putProperty(msgInner, MessageConst.PROPERTY_REAL_QUEUE_ID, String.valueOf(msgInner.getQueueId()));
        // 重置系统标记中的事务消息状态位，改为非事务消息
        msgInner.setSysFlag(MessageSysFlag.resetTransactionValue(msgInner.getSysFlag(), MessageSysFlag.TRANSACTION_NOT_TYPE));
        // 把PREPARED消息的Topic改为RMQ_SYS_TRANS_HALF_TOPIC
        msgInner.setTopic(TransactionalMessageUtil.buildHalfTopic());
        // 这样所有的PREPARED事务消息都发到同一个Topic和Queue，并且不会被Consumer消费
        msgInner.setQueueId(0);
        msgInner.setPropertiesString(MessageDecoder.messageProperties2String(msgInner.getProperties()));

        return msgInner;
    }

EndTransactionProcessor#processRequest

    @Override
    public RemotingCommand processRequest(ChannelHandlerContext ctx, RemotingCommand request) throws RemotingCommandException {
        // ...
        // Slave不接受事务消息
        if (BrokerRole.SLAVE == brokerController.getMessageStoreConfig().getBrokerRole()) {
            // ...
        }
        if (requestHeader.getFromTransactionCheck()) {
            switch (requestHeader.getCommitOrRollback()) {
                // Producer发送结束事务请求时本地事务状态处于待决Pending状态，直接返回
                case MessageSysFlag.TRANSACTION_NOT_TYPE: {
                    return null;
                }
                // Producer发送结束事务请求时本地事务状态处于Commit状态，可以继续
                case MessageSysFlag.TRANSACTION_COMMIT_TYPE: {
                    break;
                }
                // Producer发送结束事务请求时本地事务状态处于Rollback状态，可以继续
                case MessageSysFlag.TRANSACTION_ROLLBACK_TYPE: {
                    break;
                }
                default:
                    return null;
            }
        } else {
            // 同上
        }

        OperationResult result = new OperationResult();
        // Producer提交事务消息
        if (MessageSysFlag.TRANSACTION_COMMIT_TYPE == requestHeader.getCommitOrRollback()) {
            // 根据请求头携带的偏移从CommitLog获取对应的PREPARED消息
            result = this.brokerController.getTransactionalMessageService().commitMessage(requestHeader);
            if (result.getResponseCode() == ResponseCode.SUCCESS) {
                // 检查PREPARED消息
                RemotingCommand res = checkPrepareMessage(result.getPrepareMessage(), requestHeader);
                if (res.getCode() == ResponseCode.SUCCESS) {
                    // 恢复原本的消息的Topic等属性，给Consumer消费
                    MessageExtBrokerInner msgInner = endMessageTransaction(result.getPrepareMessage());
                    // ...
                    // 将恢复后的Prepared消息再次持久化到CommitLog，生成响应结果返回
                    RemotingCommand sendResult = sendFinalMessage(msgInner);
                    if (sendResult.getCode() == ResponseCode.SUCCESS) {
                        /*
                         * 为恢复后的Prepared消息生成OP消息，写入CommitLog
                         * OP消息的消息体内容为恢复Topic和QueueId的Prepared消息在ConsumeQueue的位置偏移，回查时会涉及到
                         * OP消息标记该Prepared消息的事务已结束，已经Commit或Rollback，如果Prepared消息没有OP消息，说明事务状态还未确定
                         */
                        this.brokerController.getTransactionalMessageService().deletePrepareMessage(result.getPrepareMessage());
                    }
                    return sendResult;
                }
                return res;
            }
        }
        // 回滚事务消息
        else if (MessageSysFlag.TRANSACTION_ROLLBACK_TYPE == requestHeader.getCommitOrRollback()) {
            // 根据请求头携带的偏移从CommitLog获取对应的PREPARED消息
            result = this.brokerController.getTransactionalMessageService().rollbackMessage(requestHeader);
            if (result.getResponseCode() == ResponseCode.SUCCESS) {
                // 检查PREPARED消息
                RemotingCommand res = checkPrepareMessage(result.getPrepareMessage(), requestHeader);
                // 因为PREPARED消息本来是对Consumer不可见的，所以这里是什么也不用干，因为CommitLog是顺序写入的，所以删除消息也没有意义
                if (res.getCode() == ResponseCode.SUCCESS) {
                    // 为PREPARED消息生成OP消息，写入CommitLog
                    this.brokerController.getTransactionalMessageService().deletePrepareMessage(result.getPrepareMessage());
                }
                return res;
            }
        }
        response.setCode(result.getResponseCode());
        response.setRemark(result.getResponseRemark());
        return response;
    }

    private boolean addRemoveTagInTransactionOp(MessageExt messageExt, MessageQueue messageQueue) {

    }

TransactionalMessageBridge#writeOp

    private void writeOp(Message message, MessageQueue mq) {
        MessageQueue opQueue;
        // 本地缓存已经有该QueueId和Topic的OpMessageQueue
        if (opQueueMap.containsKey(mq)) {
            opQueue = opQueueMap.get(mq);
        }
        // 没有
        else {
            // 创建一个OpMessageQueue，Topic为RMQ_SYS_TRANS_OP_HALF_TOPIC，BrokerName和QueueId和传入的一样
            opQueue = getOpQueueByHalf(mq);
            // 加入
            MessageQueue oldQueue = opQueueMap.putIfAbsent(mq, opQueue);
            // 双重检查，已存在
            if (oldQueue != null) {
                // 用已经存在的
                opQueue = oldQueue;
            }
        }
        // 加入时不存在
        if (opQueue == null) {
            // 和getOpQueueByHalf一模一样
            opQueue = new MessageQueue(TransactionalMessageUtil.buildOpTopic(), mq.getBrokerName(), mq.getQueueId());
        }
        // 将该OP消息写入CommitLog
        putMessage(makeOpMessageInner(message, opQueue));
    }

TransactionalMessageServiceImpl#check

    @Override
    public void check(long transactionTimeout, int transactionCheckMax, AbstractTransactionalMessageCheckListener listener) {
        try {
            // 半消息的Topic，RMQ_SYS_TRANS_HALF_TOPIC
            String topic = MixAll.RMQ_SYS_TRANS_HALF_TOPIC;
            // 获取Prepared消息的MessageQueue列表
            Set<MessageQueue> msgQueues = transactionalMessageBridge.fetchMessageQueues(topic);
            if (msgQueues == null || msgQueues.size() == 0) {
                return;
            }
            // 遍历Prepared消息的MessageQueue，貌似现在固定只有一个Queue？
            for (MessageQueue messageQueue : msgQueues) {
                long startTime = System.currentTimeMillis();
                /*
                 * 从本地缓存获取该Prepared消息的MessageQueue映射的Op消息的MessageQueue，没有就创建并加入，MessageQueue是有重写equals方法的
                 * Prepared消息的MessageQueue的Topic是RMQ_SYS_TRANS_HALF_TOPIC，Op消息的MessageQueue是RMQ_SYS_TRANS_OP_HALF_TOPIC
                 * RMQ_SYS_TRANS_HALF_TOPIC：Prepared消息的Topic
                 * RMQ_SYS_TRANS_OP_HALF_TOPIC：Broker收到事务消息的Commit或Rollback请求后，将消息存储在该Topic
                 */
                MessageQueue opQueue = getOpQueue(messageQueue);
                // 从本地缓存获取Prepared队列的处理偏移，没找到就使用ConsumeQueue里的最小偏移替代
                long halfOffset = transactionalMessageBridge.fetchConsumeOffset(messageQueue);
                // 从本地缓存获取Op队列的处理偏移，没找到就使用ConsumeQueue里的最小偏移替代
                long opOffset = transactionalMessageBridge.fetchConsumeOffset(opQueue);
                // 非法偏移，跳过该MessageQueue
                if (halfOffset < 0 || opOffset < 0) {
                    continue;
                }
                List<Long> doneOpOffset = new ArrayList<>();
                HashMap<Long, Long> removeMap = new HashMap<>();
                /*
                 * 根据OP消息的队列处理偏移从CommitLog获取32条OP消息
                 * 并遍历拉取到的OP消息，根据消息体获取OP消息对应的Prepared消息在队列的偏移
                 * 再根据传入的Prepared消息的队列处理偏移，填充已检查的OP消息队列偏移doneOpOffset和待移除的Prepared消息的队列偏移映射
                 */
                PullResult pullResult = fillOpRemoveMap(removeMap, opQueue, opOffset, halfOffset, doneOpOffset);
                // 只有MessageStore已关闭或不可读才返回Null，跳过该MessageQueue
                if (null == pullResult) {
                    continue;
                }
                int getMessageNullCount = 1;
                // 当前半消息队列的处理偏移，RMQ_SYS_TRANS_HALF_TOPIC@QueueId
                long newOffset = halfOffset;
                long i = halfOffset;
                while (true) {
                    // 处理时间超过60秒
                    if (System.currentTimeMillis() - startTime > MAX_PROCESS_TIME_LIMIT) {
                        break;
                    }
                    // 说明该Prepared消息已经Commit或Rollback
                    if (removeMap.containsKey(i)) {
                        removeMap.remove(i);
                        // 跳过下面的操作，不处理
                    }
                    // Prepared消息可能结束了，也可能没有
                    else {
                        // 根据Prepared队列的处理偏移从CommitLog获取Prepared消息
                        GetResult getResult = getHalfMsg(messageQueue, i);
                        MessageExt msgExt = getResult.getMsg();
                        // Prepared消息不存在
                        if (msgExt == null) {
                            // 默认重试一次
                            if (getMessageNullCount++ > MAX_RETRY_COUNT_WHEN_HALF_NULL) {
                                // 超过重试次数，直接跳出，结束该队列的事务回查
                                break;
                            }
                            // 没有新的Prepared消息，可能是处理完了
                            if (getResult.getPullResult().getPullStatus() == PullStatus.NO_NEW_MSG) {
                                // 结束该队列的事务回查
                                break;
                            } else {
                                // 偏移非法，根据校正的偏移，下个循环重新拉取
                                i = getResult.getPullResult().getNextBeginOffset();
                                newOffset = i;
                                continue;
                            }
                        }
                        // 该Prepared消息超过回查次数，方法内部有计数，需要丢弃 || 存储时间过久，需要跳过
                        if (needDiscard(msgExt, transactionCheckMax) || needSkip(msgExt)) {
                            // 把该Prepared消息Topic设置为TRANS_CHECK_MAX_TIME_TOPIC，再次写入CommitLog
                            listener.resolveDiscardMsg(msgExt);
                            // 下个循环处理下一条消息
                            newOffset = i + 1;
                            i++;
                            // 跳过该消息
                            continue;
                        }
                        // 该Prepared消息存储时间大于本次处理的开始时间，说明是在处理过程中新加入的消息
                        if (msgExt.getStoreTimestamp() >= startTime) {
                            // 结束该队列的事务回查，留给下次再检查
                            break;
                        }
                        // 该Prepared消息出生后经过的时间 = 当前时间 - 消息出生的时间戳
                        long valueOfCurrentMinusBorn = System.currentTimeMillis() - msgExt.getBornTimestamp();
                        // 检查豁免时间，因为Producer发送Prepared消息后，到发送Commit或Rollback消息结束事务这段时间内，就是事务的超时时间，这段时间不需要回查
                        long checkImmunityTime = transactionTimeout;
                        String checkImmunityTimeStr = msgExt.getUserProperty(MessageConst.PROPERTY_CHECK_IMMUNITY_TIME_IN_SECONDS);
                        // 如果Prepared消息指定了事务消息过期时间的属性
                        if (null != checkImmunityTimeStr) {
                            // 更新豁免时间
                            checkImmunityTime = getImmunityTime(checkImmunityTimeStr, transactionTimeout);
                            // 消息还没到回查的时间
                            if (valueOfCurrentMinusBorn < checkImmunityTime) {
                                // 将消息重新写入CommitLog
                                if (checkPrepareQueueOffset(removeMap, doneOpOffset, msgExt)) {
                                    // 如果写入CommitLog成功，跳过该条Prepared消息
                                    newOffset = i + 1;
                                    i++;
                                    continue;
                                }
                                // 如果写入CommitLog不成功，则不能跳过该消息
                            }
                        } else {
                            // Prepared消息没指定过期时间，同时也没到回查的时间，可能是新的消息
                            if ((0 <= valueOfCurrentMinusBorn) && (valueOfCurrentMinusBorn < checkImmunityTime)) {
                                // 结束该队列的事务回查，留给下次再检查
                                // 因为是顺序写入，所以后面的消息都是更新的消息，或者是放回的消息
                                break;
                            }
                        }
                        // 本次获取的OP消息列表
                        List<MessageExt> opMsg = pullResult.getMsgFoundList();
                        /*
                         * 判断是否需要发送事务回查消息
                         * 如果从OP队列没有拉取到消息 && Prepared消息出生后已经过了事务超时（没拉取到新的OP消息说明目前已收到的OP消息已经处理完，还没找到对应的事务结束消息）
                         * 如果从OP队列拉取到消息 && 拉取的最后一条OP消息的出生时间和本次队列处理的开始时间的时间差大于事务超时时间（暂时没看懂）
                         * 如果该Prepared消息的？，这个是在什么情况下发生？
                         */
                        boolean isNeedCheck = (opMsg == null && valueOfCurrentMinusBorn > checkImmunityTime) || (opMsg != null && (opMsg.get(opMsg.size() - 1).getBornTimestamp() - startTime > transactionTimeout)) || (valueOfCurrentMinusBorn <= -1);
                        // 如果需要发送事务回查消息
                        if (isNeedCheck) {
                            // 将该Prepared消息重新写入CommitLog
                            if (!putBackHalfMsgQueue(msgExt, i)) {
                                // 如果重新写入不成功，不能发送回查消息，下个循环继续尝试
                                continue;
                            }
                            // 如果重新写入CommitLog成功
                            // 把Prepared消息恢复原本Topic和QueueId，并根据ProducerGroup获取一个Producer
                            // 发送CHECK_TRANSACTION_STATE的单向请求给Producer回查
                            listener.resolveHalfMsg(msgExt);
                        }
                        // 不确定是否需要发送回查消息
                        else {
                            // 继续加载后32条OP消息
                            pullResult = fillOpRemoveMap(removeMap, opQueue, pullResult.getNextBeginOffset(), halfOffset, doneOpOffset);
                            // 下个循环继续检查
                            continue;
                        }
                    }
                    // 该Prepared消息已经确认事务结束或者已经发送回查消息，更新半消息队列的处理偏移
                    newOffset = i + 1;
                    i++;
                } // 该MessageQueue的Prepared消息循环处理结束
                // 说明循环中处理了消息
                if (newOffset != halfOffset) {
                    // 更新Prepared队列的回查处理偏移
                    transactionalMessageBridge.updateConsumeOffset(messageQueue, newOffset);
                }
                // 计算Op队列的处理偏移
                long newOpOffset = calculateOpOffset(doneOpOffset, opOffset);
                if (newOpOffset != opOffset) {
                    // 更新Op队列的处理偏移
                    transactionalMessageBridge.updateConsumeOffset(opQueue, newOpOffset);
                }
            }
        } catch (Exception e) {
            // ...
        }

    }

TransactionalMessageServiceImpl#fillOpRemoveMap

    private PullResult fillOpRemoveMap(HashMap<Long, Long> removeMap, MessageQueue opQueue, long pullOffsetOfOp, long miniOffset, List<Long> doneOpOffset) {
        // 根据OP消息的队列偏移从CommitLog获取32条OP消息
        // 这里的消费是指事务处理，因为消息没有Consumer，或者说Broker才是Consumer
        PullResult pullResult = pullOpMsg(opQueue, pullOffsetOfOp, 32);
        // 只有MessageStore已关闭或不可读才返回Null
        if (null == pullResult) {
            return null;
        }
        // ...
        // 遍历拉取到的OP消息
        for (MessageExt opMessageExt : opMsg) {
            // 从OP消息的消息体获取对应的Prepared消息在ConsumeQueue的位置偏移
            Long queueOffset = getLong(new String(opMessageExt.getBody(), TransactionalMessageUtil.charset));
            if (TransactionalMessageUtil.REMOVETAG.equals(opMessageExt.getTags())) {
                // 如果该Prepared消息在队列的偏移小于当前已处理的Prepared消息在队列的偏移
                if (queueOffset < miniOffset) {
                    // 说明当前OP消息已处理/检查，把队列偏移加入
                    doneOpOffset.add(opMessageExt.getQueueOffset());
                } else {
                    // 说明Prepared消息待移除
                    // Prepared消息在队列的偏移 - 对应的OP消息在队列的偏移
                    removeMap.put(queueOffset, opMessageExt.getQueueOffset());
                }
            } else {
            }
        }
        return pullResult;
    }

TransactionalMessageServiceImpl#checkPrepareQueueOffset

    private boolean checkPrepareQueueOffset(HashMap<Long, Long> removeMap, List<Long> doneOpOffset, MessageExt msgExt) {
        String prepareQueueOffsetStr = msgExt.getUserProperty(MessageConst.PROPERTY_TRANSACTION_PREPARED_QUEUE_OFFSET);
        // 这个属性，只有putImmunityMsgBackToHalfQueue将消息重新写入CommitLog后才有
        if (null == prepareQueueOffsetStr) {
            // 将消息重新写入CommitLog，返回是否写入成功
            // 这里会设置重写消息的TRAN_PREPARED_QUEUE_OFFSET属性为原本的Prepared消息在队列的偏移
            // 所以无论消息重新写入多少次，这个属性值都是最初的消息的队列偏移
            return putImmunityMsgBackToHalfQueue(msgExt);
        }
        // 消息是重新写入CommitLog的消息
        else {
            // 获取最初的消息在队列的偏移
            long prepareQueueOffset = getLong(prepareQueueOffsetStr);
            if (-1 == prepareQueueOffset) {
                return false;
            } else {
                // 说明最初的Prepared消息已经收到OP消息，事务已经结束
                if (removeMap.containsKey(prepareQueueOffset)) {
                    long tmpOpOffset = removeMap.remove(prepareQueueOffset);
                    doneOpOffset.add(tmpOpOffset);
                    // 不需要再回查
                    return true;
                }
                // 不知道事务有没有结束
                else {
                    // 将消息重新写入CommitLog，返回是否写入成功
                    return putImmunityMsgBackToHalfQueue(msgExt);
                }
            }
        }
    }

TransactionalMessageServiceImpl#putBackHalfMsgQueue

    private boolean putBackHalfMsgQueue(MessageExt msgExt, long offset) {
        // 将该半消息重新写入CommitLog
        PutMessageResult putMessageResult = putBackToHalfQueueReturnResult(msgExt);
        if (putMessageResult != null && putMessageResult.getPutMessageStatus() == PutMessageStatus.PUT_OK) {
            /*
             * 将重新写入的Prepared消息的QueueOffset、CommitLogOffset设置为重新存入的偏移，后面会发给Producer回查
             * 只要消息重新写入CommitLog成功，后面就会发送回查消息，只要发送了回查消息，进度就会向前
             * 如果回查失败，可以通过放回的Prepared消息再次发送回查消息
             * 如果回查成功，可以通过Op队列中的消息判断是否已经确认
             */
            msgExt.setQueueOffset(putMessageResult.getAppendMessageResult().getLogicsOffset());
            msgExt.setCommitLogOffset(putMessageResult.getAppendMessageResult().getWroteOffset());
            msgExt.setMsgId(putMessageResult.getAppendMessageResult().getMsgId());
            return true;
        } else {
            return false;
        }
    }

TransactionalMessageServiceImpl#calculateOpOffset

    private long calculateOpOffset(List<Long> doneOffset, long oldOffset) {
        Collections.sort(doneOffset);
        // 上次更新的Op队列的处理偏移
        long newOffset = oldOffset;
        // 遍历已经处理的Op消息的队列偏移
        for (int i = 0; i < doneOffset.size(); i++) {
            if (doneOffset.get(i) == newOffset) {
                // 可以处理下一个
                newOffset++;
            } else {
                break;
            }
        }
        return newOffset;

    }
