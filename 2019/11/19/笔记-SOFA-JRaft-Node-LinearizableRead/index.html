[笔记]-SOFA-JRaft-Node-LinearizableRead

2019-11-19

关于一致性，这篇文章讲得挺好的。

线性一致读，不能直接从当前Leader节点读，因为无法确定当前Leader仍是Leader，比如网络分区。

Raft Log

同写请求一样，读请求也走一遍Raft协议，但开销较大。

ReadIndex Read

Leader将自己当前Log的CommitIndex记录到ReadIndex。

Leader向Follower发起一轮Heartbeat，如果半数以上节点返回了对应响应，那么就能确定自己仍是Leader。

Leader等待自己的状态机执行，直到ApplyIndex超过了ReadIndex，这样就能安全的提供线性一致读，不管读的时候是否仍是Leader。

Leader执行读请求，将结果返回给Client。

其实也可以从Follower读，但是当Follower进度落后太多的情况下，响应时间会变长。

Lease Read

类似ReadIndex，但省去了Log和网络交互。

Leader取一个比ElectionTimeout小一个数量级的租期（Lease），在租约期内不会发生选举，确保了Leader不会变，所以可以略过心跳。

Wait Free

节点有两个状态，日志状态和状态机状态，Leader的日志状态表示当前集群最新的数据状态，因为所有的写请求一定都先记录在日志中。

只要保证当前时刻，当前节点状态机是最新即可，但是Leader节点的状态机不一定是最新的，比如新当选的Leader。

当Leader已经成功应用了自己任期的第一条日志后，状态机状态一定是满足线性一致读的。
