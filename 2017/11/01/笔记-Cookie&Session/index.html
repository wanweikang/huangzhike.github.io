  • 保存会话数据的两种技术
    • Cookie是客户端技术，数据以cookie给浏览器，浏览器再访问服务器会带着数据去。

删除Cookie

访问路径由资源路径与资源名称构成。默认 Cookie 与访问路径中的资源路径绑定。
只要用户发出带有绑定资源路径的请求，则在请求头部，将自动会携带与之绑定的 Cookie 数据。

访问路径 http://127.0.0.1:8080/cookie/test/some，在响应头部，携带有设置的 Cookie。
此时的两个 Cookie 默认与资源路径 http://127.0.0.1:8080/cookie/test 进行了绑定。
当在浏览器地址栏中再次输入包含以上资源路径的请求地址后，如 http://127.0.0.1:8080/cookie/test/xxx/ooo/jjj。
一旦发出请求，在请求头部已经包含了刚才由服务端发送来的Cookie 数据。

设置绑定路径

当浏览器发出 http://127.0.0.1:8080/cookie/test/some 请求后会将 Cookie 放入到响应头部，并发送给客户端。
但不会将 Cookie 与该访问路径的资源路径绑定，而是将 Cookie 与代码中指定的路径绑定。
此时浏览器再发出包含 http://127.0.0.1:8080/cookie/test 资源路径的请求时，不会携带 Cookie。
只有发出包含 http://127.0.0.1:8080/cookie/abc 资源路径的请求时，请求头才携带 Cookie。

  • 一个cookie只能标识一种信息，至少含有名（NAME）和设置值（VALUE）
  • 浏览器一般只允许存放300个cookie，每个站点最多存放20个，每个cookie大小限制为4KB
  • 默认是会话级别cookie(存储于浏览器内存)，浏览器进程关闭则删除。有效期通过maxAge设置，存于硬盘，0表示删除
  • 删除cookie时，path必须一致，否则不会删除

session对象由服务器创建,一个浏览器(会话)独占一个session对象。

  • 创建：调用request对象的getSession方法后才会创建session对象，getSession(false)只获取，不创建
  • 销毁：默认30分钟没使用则自动销毁。失效时间可在web.xml的<session-config>标签中使用<session-timeout>，单位分钟，也可调用session对象的invalidate方法销毁

session基于cookie实现，将JSESSIONID写回浏览器，这个cookie(JSESSIONID)默认没设置有效期,关浏览器即销毁。可以自行回写JSESSIONID覆盖原cookie值。

服务器接收到客户端的请求后，从请求中读取 Cookie。
若没有找到名称为 JSESSIONID 的 Cookie，则说明是新请求，服务器生成一个新的 Session，并将新的 JSSEIONID 保存到 Cookie 中发送给客户端。

浏览器禁用Cookie后的session解决方案：URL重写

URL 重写：将 JSessionID 添加到请求的最后
手工实现：将 JSessionID 添加到地址栏的请求最后。key 为全小写的 jsessionid，而 value 则为服务端响应头部发送来的 JSESSIONID。
原请求与 jsessionid 间使用分号分隔，而非问号。
http://127.0.0.1:8080/Xxx/test/some;jsessionid=42F374„„16D6D

response.encodeRedirectURL(String url)：对sendRedirect方法后的url地址重写
response.encodeURL(String url)：对表单action和超链接的url地址重写

两个方法非常智能的，当检测到浏览器没有禁用cookie时，就不进行URL重写

利用Session防止表单重复提交

客户端无法解决，就在服务器端解决，用session。
在服务器端生成唯一的随机标识号，Token(令牌)，在当前用户的Session域中保存这个Token。
然后将Token发送到客户端的Form表单中，在Form表单中用隐藏域存储Token，表单提交时连Token一起提交到服务器端。
在服务器端判断客户端提交上来的Token与服务器端生成的Token是否一致，如果不一致，是重复提交，如果相同则处理表单提交，处理完后清除当前用户的Session域中存储的标识号。

  • 下列情况下，服务器拒绝处表单请求：
    • 存储Session域中的Token(令牌)与表单提交的Token(令牌)不同。
    • 当前用户的Session中不存在Token(令牌)。
    • 用户提交的表单数据中没有Token(令牌)。
  • 不用session,用cookie,减小服务器压力
  • 若浏览器禁用cookie,需使用url自带sessionid，相关方法:encodeURL和encodeRedirectURL

  • 禁止表单重复提交：在JavaScript或服务器实现。

    • JavaScript：用户可修改js、自建表单提交，刷新页面，后退等方法重复提交；
    • 服务器实现是给每个表单一个随机表单号
    • 生成表单号使用“令牌发生器”，为保证唯一，使用单例。
    • base64编码，三字节变四字节，每6位变8位(一字节)，高位补零，每字节最大值为63。
    • md5，常用于保存密码(可能为防止破解会加随机数)，校验数据完整性

  • 三个域对象

    • Session
    • Request
    • ServletContext

  • 容器选用

    • 数据显示完了就没用了，则选用Request
    • 数据除了显示外，稍后还会用，则选用Session
    • 数据除了显示外，不仅稍后会用，还会给别人用，则选用ServletContext

如果想要关掉浏览器再打开还可以用同一个session，则需要给服务器回送的cookie设置有效时间（服务器自动回送的时候是没有有效期的）。
通过session对象的getId方法获得该session的id,然后创建一个cookie，该cookie的名字为”JSESSIONID”，值就是刚刚获得的id，再将该cookie设置下有效期，并添加到cookie中即可。
但是有效期不得超过30分钟，因为浏览器关掉后，session只保存30分钟。

服务器创建session出来后，会把session的id号以cookie的形式回写给客户机，只要浏览器不关，再去访问服务器时，都会带着session的id号去，服务器发现客户机带session的id过来了，就会使用内存中与之对应的session为之服务。

服务器第一次创建session，把session的id号手动以cookie的形式回送给浏览器，并设置cookie的有效期，这样即使浏览器关了，开新的浏览器时，还会带着session的id号找服务器，服务器从而就可以用内存中与之对应的session为第二个浏览器窗口服务。

第一次request.getSession时创建。session对象默认在30分钟没有使用，自动销毁session。
可以在web.xml文件中手动配置session的失效时间。
