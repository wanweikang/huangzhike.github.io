[笔记]-Mybatis-缓存&分页

2017-12-04 0:23

一级缓存：存储作用域为 Session，当 Session flush 或 close 之后，该 Session 所有 Cache 清空。
二级缓存：存储作用域为 Mapper(Namespace)，并且可自定义存储源，如 Ehcache。
从缓存读取数据的依据是查询 SQL 的映射 id，而非查询结果。
增删改操作，无论是否进行提交，均会清空一级、二级查询缓存。

开启二级缓存

映射语句文件中的所有 select 语句将会被缓存。
映射语句文件中的所有 insert,update 和 delete 语句会刷新缓存。
缓存会使用 Least Recently Used(LRU,最近最少使用的)算法来收回。
每个 namespace 的 mapper 都会有自己的缓存空间。
如果两个 mapper 的 namespace 相同，查询到的数据将存储到相同的二级缓存。
sqlSession 执行了commit 会清空二级缓存。

全局关闭二级缓存：整个应用的二级缓存全部关闭。

局部关闭二级缓存：只是针对于某个<select/>查询，不使用二级缓存。

若要使某个增、删或改操作不清空二级缓存，则需要在其<insert/>或<delete/>或<update/>中添加属性 flushCache=”false”。

二级缓存的使用原则

  • 只能在一个命名空间下使用二级缓存
    由于二级缓存中的数据是基于 namespace 的，即不同 namespace 中的数据互不干扰。
    在多个 namespace 中若均存在对同一个表的操作，可能出现不一致。
  • 在单表上使用二级缓存
    如果一个表与其它表有关联关系，可能存在多个 namespace 对同一数据的操作。
    而不同 namespace 中的数据互不干扰，可能出现这多个 namespace 中的数据不一致现象。
  • 查询多于修改时使用二级缓存
    在查询操作远远多于增删改操作的情况下可以使用二级缓存。
    任何增删改操作都将刷新二级缓存，对二级缓存的频繁刷新将降低系统性能。

实现 Mybatis 的二级缓存

  1. 采用 Mybatis 内置的 cache 机制
  2. 采用三方 cache 框架，如 ehcache，实体类无需实现序列化接口。

ehcache是一个分布式缓存框架

不使用分布缓存，缓存的数据在各服务单独存储，不方便系统开发。

Mybatis提供了cache接口，如果要实现自己的缓存逻辑，实现cache接口开发即可。

Mybatis和ehcache整合包中提供了cache接口的实现类。

配置mapper中cache中的type为ehcache对cache接口的实现类型

classpath:ehcache.xml

分页(pagination)

数据库的某个表里有1000条数据，每次只显示100条，第1页显示第0到第99条，这就是分页。

逻辑分页：程序在显示每页的数据时，首先查询得到表中的1000条数据，然后分页显示。

物理分页：程序先判断出该选出这1000条的第几条到第几条，然后数据库查询出需要的100条返回。

MyBatis 使用 RowBounds 实现的分页是逻辑分页，先把数据记录全部查询出来，再根据 offset 和 limit 截断记录返回。

为了在数据库层面上实现物理分页，又不改变原来 MyBatis 的函数逻辑，可以编写 plugin 截获 MyBatis Executor 的 statementhandler，重写SQL来执行查询。

对于物理分页，Mysql 利用 limit offset, pagesize 方式来实现，Oracle 是通过 rownum 来实现的。
