[笔记]-&0xff

2018-12-07

由byte & 0xff引申出来，参考这篇文章。

  • 机器数：数在计算机中的二进制表示形式，带符号，最高位存放符号，正数为0，负数为1。

    • 十进制+3，字长为8位，二进制就是0000 0011。

    • 如果是-3，就是1000 0011。

    • 这里0000 0011和1000 0011就是机器数。

  • 真值：机器数的形式值不等于真正的数值，因为第一位是符号位。

    • 有符号数1000 0011，其最高位1代表负，真正数值是-3，而不是形式值131（1000 0011转换成十进制等于131）。

    • 区别起见，将带符号位的机器数对应的真正数值称为机器数的真值。

    • 例：0000 0001的真值=+000 0001=+1，1000 0001的真值 =–000 0001=–1。

  • 原码：符号位加上真值的绝对值，即用第一位表示符号，其余位表示值。

    • [+1]十进制=[0000 0001]原码。

    • [-1]十进制=[1000 0001]原码。

    • 因为第一位是符号位，所以8位二进制数的取值范围就是:[1111 1111, 0111 1111]，即[-127, 127] 。

  • 反码：正数的反码是其本身，负数的反码是在其原码的基础上，符号位不变，其余各个位取反。

    • [+1]十进制 = [0000 0001]原码 = [0000 0001]反码。

    • [-1]十进制 = [1000 0001]原码 = [1111 1110]反码。

  • 补码:正数的补码就是其本身，负数的补码是在其原码的基础上，符号位不变，其余各位取反，最后+1(即在反码的基础上+1)。

    • [+1]十进制 = [0000 0001]原码 = [0000 0001]反码 = [0000 0001]补码。

    • [-1]十进制 = [1000 0001]原码 = [1111 1110]反码 = [1111 1111]补码。

减去一个正数等于加上一个负数，即:1 - 1 = 1 + (-1) = 0，所以可以只有加法而没有减法，这样计算机运算的设计就更简单了。

  • 1 - 1=1 + (-1) = [0000 0001]原码 + [1000 0001]原码 = [1000 0010]原码 = -2。

为了解决原码做减法的问题，出现了反码:

  • 1 - 1 = 1 + (-1) = [0000 0001]原码 + [1000 0001]原码
    = [0000 0001]反码 + [1111 1110]反码 = [1111 1111]反码
    = [1000 0000]原码 =-0

用反码计算减法，结果的真值部分是正确的，但是0带符号会有[0000 0000]原码和[1000 0000]原码表示0。

补码解决了0的符号以及两个编码的问题:

  • 1 - 1 = 1 + (-1)= [0000 0001]原码 + [1000 0001]原码
    = [0000 0001]补码 + [1111 1111]补码 = [0000 0000]补码
    = [0000 0000]原码

而且可以用[1000 0000]表示-128:

  • (-1) + (-127) = [1000 0001]原码 + [1111 1111]原码
    = [1111 1111]补码 + [1000 0001]补码 = [1000 0000]补码

在用补码运算的结果中，-128并没有原码和反码表示（对-128的补码表示[1000 0000]补算出来的原码是[0000 0000]原码，这是不正确的）。

因为第一位表示的是符号位，使用补码还能多表示一个最低数。

8位二进制，使用原码或反码表示的范围为[-127, +127]，而使用补码表示的范围为[-128, +127]。

因为机器使用补码，对于32位int类型，可以表示范围是: [-231, 230] 。

计算机内的存储都是利用二进制的补码进行存储的，位运算都是用补码进行。

无符号整数没有符号位，不能区分正负数，所以只能是正数。

Java没有unsigned无符号类型。

byte a = -127;
int b = a & 0xff; // 129

计算机系统以字节为单位，每个地址单元都对应着一个字节，1byte =8bit。

对于位数大于8位的处理器，例如16位或者32位的处理器，由于寄存器宽度大于一个字节，存在着将多个字节安排的问题。

因此就导致了大端存储模式和小端存储模式。

一个16bit的short型x，在内存中的地址为0x0010，x的值为0x1122，那么0x11为高字节，0x22为低字节。

对于大端模式，就将0x11放在低地址中，即0x0010中，0x22放在高地址中，即0x0011中。

小端模式，刚好相反。

Java和所有的网络通讯协议都是使用Big-Endian编码。

十六进制0xff的长度是一个字节，即八位，二进制为1111 1111，一个8bit数与1111 1111与运算还是这个数本身，但是一个16bit数与0xff就被截断了，比如1100 1100 1100 1100&0xff结果为1100 1100。

如果想不被截断，把0xff扩展为二个字节即0xffff，以此类推，0xffffff，0xffffffff。

4字节，32位，按照大端方式排列：

// 最高位 -> 最低位
1111 1111 1010 1010 1100 0011 1010 1010

最高位8字节要移到最低位，那么这个8个字节>> (3 * 8)，然后& 0xff，取出，后续得>> (2 * 8) & 0xff，>> (1 * 8) & 0xff均可取出。

int a转换成字节，int是4字节，需要4个byte来保存，又因为Java是大端排序，那么byte[0]为最高位，所以需要>> 24，这么一个个的把a的4个字节取出存入byte数组中，这里0xff不仅截断，而且还将有符号转换成了无符号。

将字节转换回去不是截断而是融合，需要将&改为|，还得把每个字节移到所在实际位置，比如byte[0]是最高位，因此还得将其移到4个字节的头部即需要<< 24，以此类推。

一个int32型就出现了。

但是Java在位移操作之前不能少了有符号转无符号操作。

int a = 1234567890;

byte[] b1 = new byte[4];

b1[0] = (byte) ((a >> 24) & 0xff);
b1[1] = (byte) ((a >> 16) & 0xff);
b1[2] = (byte) ((a >> 8) & 0xff);
b1[3] = (byte) (a & 0xff);

int b2 = ((b1[0] & 0xff) << 24) | ((b1[1] & 0xff) << 16) | ((b1[2] & 0xff) << 8) | (b1[3] & 0xff);

int b3 = ((b1[0]) << 24) | ((b1[1]) << 16) | ((b1[2]) << 8) | (b1[3]);

System.out.println(b2); // 正确结果
System.out.println(b3); // 错误结果，没有将有符号转无符号

  • 无符号位移：不需要补符号位

  • 有符号位移：要补符号位

正数

4 >>> 2：0000 ... 0100 -> 0000 ... 0001 = 1

4 >> 2：0000 ... 0100 -> 0000 ... 0001 = 1

负数

-4 >>> 2：1000 0000 ... 0100（原码） -> 1111 1111 ... 1011（反码）-> 1111 1111 ... 1100（补码）-> 0011 1111 ... 1111（位移，补高位） = 1073741823

-4 >> 2：1000 0000 ... 0100（原码） -> 1111 1111 ... 1011（反码）-> 1111 1111 ... 1100（补码）-> 1111 1111 ... 1111（位移，补高位）-> 1000 0000 ... 0001（原码） = -1

当需要移位的数为正数时，有符号位移（>>）和无符号位移（>>>）是相同的。
