[笔记]-&0xff

2018-12-07 20:28

由byte & 0xff引申出来，参考这篇文章：https://www.cnblogs.com/zhangziqiu/archive/2011/03/30/ComputerCode.html

  • 机器数：数在计算机中的二进制表示形式, 带符号，最高位存放符号, 正数为 0 , 负数为 1 。

    • 十进制 +3 ，计算机字长为 8 位，二进制就是 0000 0011 。

    • 如果是 -3 ，就是 1000 0011 。

    • 这里 0000 0011 和 1000 0011 就是机器数。

  • 真值：机器数的形式值不等于真正的数值，因为第一位是符号位。

    • 有符号数 1000 0011，其最高位 1 代表负，真正数值是 -3，而不是形式值 131（1000 0011转换成十进制等于131）。

    • 区别起见，将带符号位的机器数对应的真正数值称为机器数的真值。

    • 例：0000 0001 的真值 = +000 0001 = +1，1000 0001 的真值 = –000 0001 = –1 。

  • 原码：符号位加上真值的绝对值, 即用第一位表示符号, 其余位表示值。

    • [+1]十进制 = [0000 0001]原码。

    • [-1]十进制 = [1000 0001]原码。

    • 因为第一位是符号位, 所以 8 位二进制数的取值范围就是:[1111 1111 , 0111 1111] 即 [-127 , 127] 。

  • 反码：正数的反码是其本身，负数的反码是在其原码的基础上, 符号位不变，其余各个位取反。

    • [+1]十进制 = [0000 0001]原码 = [0000 0001]反码。

    • [-1]十进制 = [1000 0001]原码 = [1111 1110]反码。

  • 补码:正数的补码就是其本身，负数的补码是在其原码的基础上, 符号位不变, 其余各位取反, 最后 +1 (即在反码的基础上 +1)。

    • [+1]十进制 = [0000 0001]原码 = [0000 0001]反码 = [0000 0001]补码。

    • [-1]十进制 = [1000 0001]原码 = [1111 1110]反码 = [1111 1111]补码。

减去一个正数等于加上一个负数, 即: 1 - 1 = 1 + (-1) = 0 ,所以可以只有加法而没有减法, 这样计算机运算的设计就更简单了。

  • 1 - 1 = 1 + (-1) = [0000 0001]原码 + [1000 0001]原码 = [1000 0010]原码 = -2。

为了解决原码做减法的问题, 出现了反码:

  • 1 - 1 = 1 + (-1) = [0000 0001]原码 + [1000 0001]原码
    = [0000 0001]反码 + [1111 1110]反码 = [1111 1111]反码
    = [1000 0000]原码 = -0

用反码计算减法, 结果的真值部分是正确的, 但是 0 带符号会有[0000 0000]原码和[1000 0000]原码表示 0。

补码解决了 0 的符号以及两个编码的问题:

  • 1 - 1 = 1 + (-1) = [0000 0001]原码 + [1000 0001]原码
    = [0000 0001]补码 + [1111 1111]补码 = [0000 0000]补码
    = [0000 0000]原码

而且可以用[1000 0000]表示 -128:

  • (-1) + (-127) = [1000 0001]原码 + [1111 1111]原码
    = [1111 1111]补码 + [1000 0001]补码 = [1000 0000]补码

在用补码运算的结果中, -128 并没有原码和反码表示(对-128的补码表示[1000 0000]补算出来的原码是[0000 0000]原码, 这是不正确的)。

因为第一位表示的是符号位，使用补码还能多表示一个最低数。

8 位二进制, 使用原码或反码表示的范围为[-127, +127], 而使用补码表示的范围为[-128, +127]。

因为机器使用补码, 对于 32 位 int 类型, 可以表示范围是: [-231, 230] 。

计算机内的存储都是利用二进制的补码进行存储的，位运算都是用补码进行。

Java 没有 unsigned 类型。

byte -> int :将byte的内存空间高位补 1 （按符号位补位）扩充到 32 位，虽然将补码转为原码后十进制表示的值不变，但是二进制是改变了的。

计算机系统以字节为单位，每个地址单元都对应着一个字节，1 byte = 8bit。

对于位数大于 8 位的处理器，例如 16 位或者 32 位的处理器，由于寄存器宽度大于一个字节，存在着将多个字节安排的问题。

因此就导致了大端存储模式和小端存储模式。

一个 16 bit 的 short 型 x，在内存中的地址为 0x0010，x 的值为 0x1122 ，那么 0x11 为高字节，0x22 为低字节。

对于大端模式，就将 0x11 放在低地址中，即 0x0010 中，0x22 放在高地址中，即 0x0011 中。

小端模式，刚好相反。

Java 和所有的网络通讯协议都是使用 Big-Endian 编码。

十六进制 0xff 的长度是一个字节，即八位，二进制为 1111 1111，一个 8 bit 数与 1111 1111 与运算还是这个数本身，但是一个 16 bit 数与 0xff 就被截断了，比如 1100 1100 1100 1100 & 0xff 结果为 1100 1100。

如果想不被截断，把 0xff 扩展为二个字节即0xffff，以此类推，0xffffff, 0xffffffff。

4 字节 ，32 位，按照大端方式排列：

最高位 8 字节要移到最低位，那么这个 8 个字节 >> (3 * 8)，然后 & 0xff，取出，后续得 >> (2 * 8) & 0xff， >> (1 * 8) & 0xff均可取出。

int a 转换成字节，一般情况下，int 4 字节，需要 4 个 byte 来保存，又因为 Java 是大端排序，那么 byte[0]为最高位，所以需要 >> 24，这么一个个的把 a 的 4 个字节取出存入 byte 数组中，这里 0xff 不仅截断，而且还将有符号转换成了无符号。

将字节转换回去不是截断而是融合 ，需要将 & 改为 |，还得把每个字节移到所在实际位置，比如 byte[0] 是最高位，因此还得将其移到 4 个字节的头部即需要 << 24 ，以此类推。

一个 int32 型就出现了。

但是 Java 在位移操作之前不能少了有符号转无符号操作。
