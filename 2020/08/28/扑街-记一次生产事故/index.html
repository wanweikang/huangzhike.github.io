[扑街]-记一次生产事故

2020-08-28

没想到我也会有写这篇文章的一天，不知道怎么形容此刻的心情。

背景：

公司周年庆，要搞一个助力返现活动，就是一个人可以发起一场助力，然后转发给别人，别人可以帮你助力，你就会收到一笔钱。

这个需求的目的是为了拉新用户，是比较赶的（哪次不赶呢），主要的要求是这样的：

  • 用户可以发起一场自己的助力活动；
  • 每邀请一位新用户助力，可以得X元（新用户指的是在活动期间注册，并且没有参与过助力活动）；
  • 每邀请一位老用户助力，可以得X元；
  • 每累计邀请X名新用户助力，可以得X元；
  • 每位用户现金奖励上限为X元；
  • 每场助力活动只允许老用户助力X次；
  • 每位用户在一个助力活动中只能助力一次；
  • 每位用户单日助力上限为X次；

于是数据库会加两张表，结构大概是：

  • 助力活动表activity：

    • activity_id：主键；
    • starter_uid：发起人用户Id；
    • gain_amount：已经获得的金额；
    • limit_amount：最大获得的金额限制；
    • status：完成状态；
    • new_user_count：新用户助力次数；
    • old_user_count：老用户助力次数；

  • 助力活动记录表record：

    • record_id：主键；
    • activity_id：所属的助力活动Id；
    • starter_uid：所属的助力活动发起人用户Id；
    • relayer_uid：助力人的用户Id；
    • relay_amount：助力金额；
    • new_user：助力人是否是新用户；
    • type：该项助力是否是累积邀请奖励；
    • drawn：是否已打款；

看出问题了么？你大概会说，在一个很火爆的活动中，加事务和SELECT ... FOR UPDATE，是很危险的，事务可能会拖太长，但是，实际上一个助力活动下的竞争不会太激烈，而这里通过主键SELECT ... FOR UPDATE只会锁定该行记录，快速提交快速释放，不会有太大问题，我本地压测的结果也是这样的。

项目上线几个小时后，突然收到告警，说是聚合层的CPU负载飙到了300%，开始没太在意，后来有同事说，钱包服务的负载很高（就是打款的RPC服务），我马上警觉，不会吧。。。

一问，钱包服务提供的打款接口是依赖外部的微信接口的，并且微信有限流，但是助力和打款是同步的。

我特么。。。

然后一查，基本上RPC调用链路基本都阻塞到助力那里了，因为有大量的打款请求阻塞，结果就是，工作线程池大概要满了吧。。。

然后发现，OpenFeign的RPC超时时间设置是一个极大的数字（大概是为了避免导表的接口时间太长超时吧，为什么不是异步导表呢！），所以没有超时熔断。

而且助力接口居然没加限流！好吧，是我的锅，当时想的是应该没关系吧。。。

我特么。。。

算了，和同事讨论了一会，先紧急拉个hotfix分支加上限流，然后修改助力逻辑，最重要的是打款必须改成异步的（调用居然接近1s，但是这个只能在生产环境测试），两种解决方式，一种是通过消息队列异步通信，另一种是通过定时任务异步打款，因为这种方式是最快并且改动最小的，所以最终决定使用定时任务的方式。

修改后的助力逻辑是这样的，还加了一个全局的每日最大打款额度需求：

首先去掉事务了，不需要100%的一致性。

SELECT * FROM activity WHERE activity_id = x;

// 从Redis获取该用户的助力次数，判断是否为新用户

// 在Redis中incr记录该用户当天的助力次数，incr记录该用户在该助力下的助力次数

try {
    // 检查该用户当天的助力次数，抛异常

    // 计算实际的助力金额relay_amount，不大于该行activity的limit_amount-gain_amount

    // 先在Redis中incrBy累积记录全局的打款金额，检查大于限制则抛异常，因为需要转为整数，所以都乘了100

    // 如果不是新用户，先在Redis中incr累积记录该助力下的老用户助力次数，检查大于限制则抛异常

    // 先在Redis中setnx记录该用户在该助力下的记录，存在则抛异常

    // 尝试更新activity表，成功则插入一条record，否则回滚Redis记录并抛异常
    // UPDATE activity SET 
    // gain_amount = gain_amount + #{relay_amount},
    // status = CASE WHEN gain_amount >= limit_amount THEN 1 ELSE 0 END, 
    // old_user_relay_count = ... , new_user_relay_count = ...
    // WHERE activity_id = #{activity_id} and limit_amount >= gain_amount + #{relay_amount}
    // AND #{oldUserLimit} >= old_user_relay_count + 1

    // 如果更新成功，插入一条record
    // INSERT INTO record ...;

    // 否则如果更新失败，回滚Redis计数并抛异常
    // 在Redis中decrBy回滚记录全局的打款金额
    // 如果不是新用户，在Redis中decr回滚该助力下的老用户助力次数
    // 在Redis中del回滚该用户在该助力下的记录
    // 抛异常

    // 如果是新用户，先在Redis中incr记录该助力下的新用户助力次数，如果取模没有余数，则进行额外的奖励
    // SELECT * FROM activity WHERE activity_id = x;
    // 计算实际的助力金额relay_amount，不大于该行activity的limit_amount-gain_amount
    try {
        // 先在Redis中incrBy累积记录全局的打款金额，检查大于限制则抛异常，因为需要转为整数，所以都乘了100

        // 尝试更新activity表，成功则插入一条record，否则回滚Redis记录并抛异常
        // UPDATE activity SET 
        // gain_amount = gain_amount + #{relay_amount},
        // status = CASE WHEN gain_amount >= limit_amount THEN 1 ELSE 0 END, 
        // old_user_relay_count = ... , new_user_relay_count = ...
        // WHERE activity_id = #{activity_id} and limit_amount >= gain_amount + #{relay_amount}
        // AND #{oldUserLimit} >= old_user_relay_count + 1

        // 如果更新成功，插入一条record
        // INSERT INTO record ...;

        // 否则如果更新失败，回滚Redis计数并抛异常
        // 在Redis中decrBy回滚记录全局的打款金额
        // 抛异常
    } catch (Exception e) {
        // 忽略
    }

} catch (Exception e) {
    // 回滚Redis的该用户当天的助力次数，Redis的decr指令自减
    // 抛异常
}

修改后的方式使用Redis拦截了一层流量，还用来进行条件判断，使得直接落到数据库的流量少了很多。

并且使用了类似CAS的乐观更新机制，原子性由数据库的锁保证，将并发下的重试逻辑交给了用户。

而异步打款的定时任务逻辑大概是：

// 先根据标记位canSubmit判断是否能继续提交任务，否则直接返回
if (!canSubmit) {
    return;
}

// 先从数据库中取出一批未打款的记录
// SELECT * FROM record WHERE draw = 0 LIMIT #{batchSize}

for (Record record : recordList) {
    try {
        EXECUTOR_SERVICE.submit(() -> {
            try {
                // 先尝试更新该项为已打款，成功则继续进行，失败则跳过
                //  UPDATE `record SET draw = 1 WHERE record_id = #{recordId} and draw = 0;
                try {
                    // PRC调用打款接口
                } catch (Exception e) {
                    // 打款失败尝试回滚
                    //  UPDATE record SET draw = 0 WHERE record_id = #{recordId} and draw = 1;
                }
            } catch (Exception e) {
            }
            // 如果线程池的工作队列数量在限制内，那么可以继续提交
            if (((ThreadPoolExecutor) EXECUTOR_SERVICE).getQueue().size() < MAX_LIMIT) {
                canSubmit = true;
            }
        });
        canSubmit = true;
    } catch (Exception e) {
        // 线程池拒绝，不能继续提交
        canSubmit = false;
    }
}

也是类似抢锁的逻辑，抢到锁（更新成功）才能打款。
